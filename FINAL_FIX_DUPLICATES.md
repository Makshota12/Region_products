# 🔧 ОКОНЧАТЕЛЬНОЕ ИСПРАВЛЕНИЕ ДУБЛИКАТОВ КРИТЕРИЕВ

## ❌ Проблема

При создании новой сессии оценки критерии дублировались:
- Создавали 1 сессию → было 11 критериев ✅
- Создавали 2 сессии → было 44 критерия ❌ (вместо 22)
- Создавали 3 сессии → было 99 критериев ❌ (вместо 33)

**Причина:** Сигнал `post_save` срабатывал неправильно или несколько раз.

---

## ✅ Решение

### 1. Полностью переписан сигнал

**Файл:** `backend/.../signals.py`

**Основные изменения:**

#### a) Добавлено логирование
```python
import logging
logger = logging.getLogger(__name__)
```
Теперь можно отслеживать, когда и сколько критериев создается.

#### b) Улучшена проверка условий
```python
# Только при создании новой сессии и не при загрузке фикстур
if not created or kwargs.get('raw', False):
    return
```

#### c) Проверка существующих критериев
```python
existing_count = AssignedCriterion.objects.filter(
    evaluation_session=instance
).count()

if existing_count > 0:
    logger.info(f"Сессия {instance.id} уже имеет {existing_count} критериев. Пропускаем создание.")
    return
```

#### d) Использование `get_or_create` вместо `bulk_create`
```python
with transaction.atomic():
    created_count = 0
    for criterion in criteria:
        _, was_created = AssignedCriterion.objects.get_or_create(
            evaluation_session=instance,
            criterion=criterion,
            defaults={
                'assigned_to': instance.created_by,
                'is_verified': False
            }
        )
        if was_created:
            created_count += 1
```

**Преимущества:**
- ✅ `get_or_create` гарантирует отсутствие дубликатов
- ✅ Транзакция обеспечивает атомарность
- ✅ Логирование помогает отслеживать процесс
- ✅ Счетчик показывает реально созданные критерии

## 📊 Техническая информация

### Модель AssignedCriterion
```python
class Meta:
    unique_together = ('evaluation_session', 'criterion')
```

**Защита на уровне БД:**
- Невозможно создать две одинаковые пары (сессия + критерий)
- PostgreSQL выдаст ошибку при попытке дублирования

### Использование get_or_create
```python
obj, created = AssignedCriterion.objects.get_or_create(
    evaluation_session=session,
    criterion=criterion,
    defaults={'assigned_to': user}
)
```

**Поведение:**
- Если запись существует → возвращает её, `created=False`
- Если записи нет → создает, `created=True`
- Атомарная операция (защита от race conditions)

### Логирование
```python
logger.info(f"Создание {criteria.count()} критериев для сессии {instance.id}")
logger.info(f"Успешно создано {created_count} критериев для сессии {instance.id}")
```

**Проверка логов:**
```bash
# В консоли Django сервера увидите:
Создание 11 критериев для сессии 7
Успешно создано 11 критериев для сессии 7
```

## 🎯 Ключевые изменения в коде

### 1. signals.py

**Было:**
```python
AssignedCriterion.objects.bulk_create(
    assigned_criteria_to_create,
    ignore_conflicts=True
)
```

**Стало:**
```python
for criterion in criteria:
    _, was_created = AssignedCriterion.objects.get_or_create(
        evaluation_session=instance,
        criterion=criterion,
        defaults={...}
    )
```

**Почему лучше:**
- ✅ Явная проверка существования
- ✅ Точный подсчет созданных записей
- ✅ Лучшая отладка
- ✅ Полная защита от дубликатов

### 2. Добавлена проверка перед созданием

**Новая логика:**
```python
existing_count = AssignedCriterion.objects.filter(
    evaluation_session=instance
).count()

if existing_count > 0:
    return  # Уже есть критерии, ничего не делаем
```

### 3. Использование транзакций

**Зачем:**
```python
with transaction.atomic():
    # Все операции или все вместе, или ничего
    for criterion in criteria:
        AssignedCriterion.objects.get_or_create(...)
```

**Преимущества:**
- Если ошибка в середине → откат всех изменений
- Защита целостности данных
- Изоляция от параллельных запросов
